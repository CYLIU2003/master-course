# `master-course` 改修に関する agent 向け実装指示書

## 件名

**master-course の Tokyu Bus データ基盤・GTFS 連携・UI/BFF 差分解消・配車前処理再構築**

---

## 0. 最優先の理解

今回の作業は「新機能追加」ではなく、**Tokyu Bus 系のデータ基盤を GTFS 中心に作り直し、UI / BFF / backend / research core を再接続して、実際にシミュレーション可能な状態へ戻すための再設計・再実装**として扱うこと。

現状 `master-course` README 上では、重い ODPT / GTFS 更新処理は通常の frontend / BFF 起動から切り離し、保存済み snapshot 優先・`forceRefresh=true` のときだけ refresh 実行、Tokyu Bus は `Raw ODPT -> Raw Archive -> Canonical JSONL -> GTFS + Sidecar -> Research Features` の layered pipeline を標準にする方針が既に明示されている。つまり今回やるべきことは、**方針の新規提案ではなく、既存方針と実装のズレを埋めること**である。加えて、`MKuranowski/TokyoGTFS` は Tokyu Bus を含む “GTFS 未提供・ODPT JSON はある” 事業者向けの GTFS 生成実装だが、バス出力は作者自身が “very bare-bone” と説明しており、しかも GPLv3 である。したがって **設計・マッピング思想の参照元としては重要だが、安易なコード流用は不可**。さらに、先生の報告書でも、ODPT 生 JSON を研究本体へ直接読ませず、取得・正規化・DB / Parquet 化・キャッシュ・ID 事前解決を分離すべきことが強く推奨されている。  ([master-course README](https://github.com/CYLIU2003/master-course), [TokyoGTFS](https://github.com/MKuranowski/TokyoGTFS))

---

## 1. 目的

この修正の目的は次の 5 つ。

1. **通常起動時に重い再読込を起こさないこと**
2. **Tokyu Bus は ODPT 生 JSON 直読ではなく、GTFS 系スナップショット中心で運用すること**
3. **路線・停留所・trip・stop_times・stop timetable・calendar / service を一気通貫で結び直すこと**
4. **営業所・車両・路線の割当 UI と API を完成させること**
5. **研究用シミュレーション計算へ流し込める粒度まで、route family / pattern / trip / deadhead / day-type を正しく整理すること**

---

## 2. 現状の問題認識

添付画像と現行挙動から、少なくとも以下の問題がある。

### 2.1 起動・再読込周り

* 一度読み込んだあとでも、ページ内 F5 で再び起動読み込み画面が出る
* 通常閲覧と重い refresh / import が分離されていない体感になっている
* README に記載の「保存済み snapshot 優先」「通常起動では refresh 前提にしない」が守られていない状態として扱うこと ([master-course README](https://github.com/CYLIU2003/master-course))

### 2.2 メイン画面の導線不整合

* いまだに「ODPT クイック取込」が残っている
* Tokyu 側は今後 **GTFS 系取り込み** を表導線にすべき
* ODPT 生取得は updater / pipeline 側に隠し、通常 UI に見せない

### 2.3 路線割当ワークフロー未完成

* 「路線未定義でバスネットワークの路線を作成してください」と出るのに、営業所へ路線を割り振る場所がない
* README には `DepotRoutePermission` と `VehicleRoutePermission` が明記されているため、**設計はあるのに UI / API が未完**という認識で直すこと ([master-course README](https://github.com/CYLIU2003/master-course))

### 2.4 公開情報エクスプローラーのデータ崩れ

* 同一系統名が大量に並ぶ
* routes と stops / trips / earliest / latest が連携していない
* 一覧上で `stops=0` や `first/last=-` が多発しており、route header と actual trip / stop_times の紐付けに失敗している
* `東98` のような例で重複が大量に見えるのは、少なくとも以下が route 一覧にそのまま混ざっている可能性が高い

  * 本線上り / 下り
  * 区間便
  * 入出庫便
  * service_id 別の断片
    つまり **route family / pattern / trip / service の階層分離に失敗**している

### 2.5 研究計算に使えない

* backend と frontend でデータモデルがズレている
* route network が simulation 用の trip graph に落ちていない
* Tokyu 側の ODPT → GTFS が未成熟で、Toei GTFS より大きく劣る
* 平日 / 土曜 / 休日の識別も route 表現に反映されていない

## 3. 設計原則

### 3.1 Tokyu Bus は「ODPT 直読 UI」から卒業する

Tokyu Bus については、**通常運用は必ず GTFS ベース**に寄せること。
ODPT は raw acquisition 用の upstream に限定し、アプリ本体は以下だけを見る。

* `GTFS/TokyuBus-GTFS/`
* `feed_metadata.json`
* `validation_report.json`
* sidecar 群
* research feature store

この repo 自体も Tokyu Bus を `Raw ODPT -> Raw Archive -> Canonical JSONL -> GTFS + Sidecar -> Research Features` の layered pipeline として定義しているので、その方針を実装側で徹底すること ([master-course README](https://github.com/CYLIU2003/master-course))

### 3.2 TokyoGTFS は「参照実装」、ただし clean-room

`TokyoGTFS` は Tokyu Bus 向け GTFS 生成の重要な参照元だが、バス出力は bare-bone で、かつ GPLv3。したがって以下のルールとする。

* **アルゴリズム・概念・データ解釈は参考にする**
* **コードの直接コピーは禁止**
* 必要なら「比較検証用 fixture」扱いにとどめる
* `README` や生成物の観察から、route grouping・calendar 化・stop_times 生成・ODPT 項目対応を学ぶ
* master-course へは clean-room 再実装する ([TokyoGTFS](https://github.com/MKuranowski/TokyoGTFS))

### 3.3 先生の報告書は全面採用でよい

報告書の内容は原則すべて適用してよい。特に以下は必須。

* 取得と研究本体の分離
* JSON-LD の正規化 / フラット化
* SQLite / DuckDB / Parquet 等の中間ストア活用
* キャッシュとバージョン管理
* `owl:sameAs` 等の ID 事前解決
* 生 JSON の都度全件ロード廃止

## 4. 最終ゴール

### 4.1 UX

* F5 しても毎回フル boot splash を出さない
* 通常閲覧は snapshot restore 優先
* heavy import / refresh は別操作
* Public Data Explorer が route family 単位で見やすい

### 4.2 データ

* Tokyu GTFS が Toei GTFS と同じ「消費者側 API 契約」で読める
* route / stop / trip / stop_times / calendar / stop timetable が整合
* `feed_id` scoped ID で Toei / Tokyu 混線なし ([master-course README](https://github.com/CYLIU2003/master-course))

### 4.3 計画

* 営業所作成 → 車両作成 → 路線 family 生成 → 営業所割当 → 車両割当 → trip graph 生成
  の一連導線が通る
* 営業所や路線 subset を選んで simulation に流せる

### 4.4 研究

* 本線・区間便・入出庫便・営業所起終点が区別される
* 平日 / 土曜 / 休日が service として分離される
* deadhead を passenger route 一覧に混ぜず、運用グラフ側では扱える

## A. 起動・再読込の改修

### A-1. BootSplash の発火条件を修正

現状の boot splash は「初回ロード」ではなく「F5 のたび」に見えている。
以下に変更すること。

* `scenario_id + snapshot_id + dataset_version + selected_feed_set` をキーにした boot state を保持
* `sessionStorage` または `IndexedDB` に次を保存

  * `last_loaded_manifest`
  * `warm_cache_timestamp`
  * `explorer_summary_cache`
  * `planning_master_cache`
* F5 時は以下を満たすなら **restore mode** に入る

  * manifest 不変
  * snapshot 不変
  * schema version 不変
* restore mode では全画面 overlay を出さず、**軽量 skeleton + silent revalidate** のみにする
* full boot splash は以下だけで表示する

  * 初回 scenario open
  * snapshot 差し替え
  * manual refresh
  * schema migration 後
  * cache missing / corruption

### A-2. `forceRefresh=true` の扱いを厳密化

README 方針どおり、通常起動では refresh しない。
app 側で refresh するのは明示操作のみ。
起動時には `/summary` 系 API だけ読みにいき、catalog refresh を絶対に自動発火しない ([master-course README](https://github.com/CYLIU2003/master-course))

---

## B. メイン画面の「ODPT クイック取込」廃止

### B-1. 表示文言を置換

メイン画面の「ODPT クイック取込」は廃止し、次のような導線へ変えること。

* **Tokyu GTFS 取り込み**
* **Public Feed Snapshot 取り込み**
* **GTFS / Sidecar 同期**
* **Raw ODPT 更新は updater ツールから実行**

### B-2. UI 上の責務分離

main app に置くのは下記のみ。

* 既存 snapshot 一覧
* GTFS import
* sidecar import / validate
* import job status
* validation report 確認

置かないもの。

* 生 ODPT の chunk fetch 実行
* raw bundle build
* stopTimetable massive refresh

これは `catalog_update_app.py` / `tools.fast_catalog_ingest` / pipeline CLI 側に閉じ込めること。README でも heavy update は main app から分離する方針なので、それを UI に反映すること ([master-course README](https://github.com/CYLIU2003/master-course))

## C. Tokyu ODPT → GTFS 生成の再設計

### C-1. 生成物の階層を明確化

Tokyu の pipeline は必ず以下の層で分けること。

1. `raw/`
2. `archive/`
3. `canonical/`
4. `gtfs/`
5. `sidecar/`
6. `features/`

### C-2. canonical 層でやること

ODPT JSON-LD から、少なくとも以下を正規化する。

* operator
* route / route family candidate
* route pattern
* stop pole
* timetable header
* timetable row
* stop timetable
* service / day-type
* depot / garage candidate
* shape candidate
* deadhead candidate
* trip operating window

ここで **JSON-LD の生構造を frontend / BFF に見せない**。
報告書どおり、深いネストは canonical 段階でフラットにする。 

### C-3. GTFS 層でやること

生成するのは最低限次。

* `agency.txt`
* `routes.txt`
* `trips.txt`
* `stop_times.txt`
* `stops.txt`
* `calendar.txt`
* 必要に応じて `calendar_dates.txt`
* `shapes.txt`（可能なら）
* `feed_info.txt` または sidecar `feed_metadata.json`

ただし Tokyu は bare-bone になりやすいので、**GTFS だけで表現しきれない研究用属性は sidecar に逃がす**。
TokyoGTFS をそのまま目標品質にせず、master-course 側で sidecar を厚くして補うこと ([TokyoGTFS](https://github.com/MKuranowski/TokyoGTFS))

### C-4. sidecar 層で保持すべき情報

必須 sidecar を追加すること。

* `route_family_map.csv` / `route_family_map.json`
* `pattern_role_map.json`
* `trip_annotation.json`
* `service_profile.json`
* `depot_candidate_map.json`
* `deadhead_link_map.json`
* `stop_area_groups.json`
* `shape_reconstruction_report.json`
* `manual_override_log.json`

中身の例。

* `route_family_id`
* `public_route_code`
* `direction_bucket`
* `pattern_role`

  * `trunk_outbound`
  * `trunk_inbound`
  * `short_turn_outbound`
  * `short_turn_inbound`
  * `pull_out`
  * `pull_in`
  * `deadhead`
  * `unknown`
* `is_revenue_trip`
* `service_ids`
* `depot_candidates`
* `display_priority`

### C-5. route family の考え方

公開情報エクスプローラーでは、ODPT 上の pattern 単位や timetable 断片を route として見せてはいけない。
**表示の主キーは route family** にすること。

例: `東98`

* family は 1 件
* その下に方向別 pattern 群
* さらにその下に service / day-type 別 trip 集計

つまり、

* 一覧表: family 単位
* 詳細ドリルダウン: pattern 単位
* 実計算: trip 単位

という 3 層に分けること。

### C-6. 平日・土曜・休日の切り分け

service の扱いを厳密化すること。

* `WEEKDAY`
* `SAT`
* `SUN_HOL`

少なくともこのレベルは分ける。
可能なら例外日は `calendar_dates.txt` または sidecar に逃がす。
**同一路線名が大量に見える原因の一部が service 混線の可能性が高い**ので、UI 表示でも service 別バッジを持たせること。

## D. route / stop / timetable / stop timetable の再接続

### D-1. route summary の算出元を見直す

現状 explorer 上で `stops=0`, `first=-`, `last=-` が大量に出るのは、route summary が route header からだけ作られている可能性が高い。
必ず以下から再計算すること。

* route family

  * 参照する pattern 群

    * 参照する trip 群

      * 参照する stop_times
      * 参照する service
      * 参照する stop timetable

### D-2. summary の定義

family summary は最低限こう求める。

* `stops`: 代表 pattern あるいは最多 `stop_count` pattern
* `trips`: family 配下の revenue trip 総数
* `first`: family 配下 trip の最早出発
* `last`: family 配下 trip の最終到着または最終出発
* `service_set`: 含まれる `service_id` 集合
* `direction_count`
* `pattern_count`
* `deadhead_count`

### D-3. stop timetable との対応

stop timetable は route family 一覧の集約元ではなく、**trip / stop_times と整合チェックする補助データ**として使うこと。
つまり主系列は必ず

`route family -> pattern -> trip -> stop_times`

であり、

`stop timetable`

は検証・補完・停留所別ビューに使う。

### G-1. feed scoped ID の徹底

* `scoped_route_id`
* `scoped_trip_id`
* `scoped_stop_id`
* `scoped_service_id`

repo でも stable `feed_id` を定義しているので、UI / BFF / backend すべてこれに合わせること ([master-course README](https://github.com/CYLIU2003/master-course))

### G-2. 共通表示モデル

Toei と Tokyu で表示 API を共通化する。
差分は feed adapter 側で吸収すること。

* Toei: official GTFS
* Tokyu: ODPT → GTFS + sidecar

UI は「どちらの feed でも同じ画面」で動くべき。

## H. simulation 計算に接続できる形へ

### H-1. route family と simulation の境界を明確化

simulation に渡すのは最終的に trip ベースだが、
ユーザーが操作する単位は基本 route family / depot / vehicle でよい。

変換は次の順で行う。

`selected depots`
→ `allowed route families`
→ `eligible patterns`
→ `revenue trips`
→ `deadhead candidates`
→ `connection graph`
→ `duty / block / dispatch problem`

### H-2. deadhead の扱い

入出庫便や回送は passenger route 一覧からは外す。
ただし simulation では必要なので、

* `is_revenue_trip = false`
* `deadhead_type = pull_out / pull_in / internal_transfer`

として残すこと。

### H-3. subset 計算

全路線で毎回計算しない前提に対応すること。
したがって、

* 営業所単位選択
* その中の route family 絞込
* 必要なら pattern 絞込

で simulation に流せる必要がある。
この要件は必須。

## 8. 受け入れ条件

以下を全部満たしたら合格。

1. 同一 scenario / 同一 snapshot で F5 しても full boot splash を毎回出さない
2. main 画面に「ODPT クイック取込」が残っていない
3. Tokyu feed は GTFS / sidecar 起点で explorer 表示される
4. route family 一覧で `東98` のような重複が統合される
5. route 行で `stops / trips / first / last` が大半の有効系統で埋まる
6. 平日 / 土曜 / 休日が service として区別される
7. 営業所 → 路線、車両 → 路線の割当 UI が存在する
8. 割当済 subset だけで simulation 入力を生成できる
9. Toei GTFS と Tokyu GTFS が同じ API モデルで閲覧できる
10. heavy refresh は updater 側でのみ行い、通常 app 起動で勝手に走らない

---

## 9. 禁止事項

* TokyoGTFS のコードをそのままコピーしない
* ODPT 生 JSON を frontend に直渡ししない
* route pattern をそのまま public route 一覧の主単位にしない
* deadhead を一般路線一覧へそのまま混ぜない
* F5 のたびに refresh / import を疑似再実行しない
* Toei / Tokyu で raw ID をそのまま共用しない

## 10. 先生の報告書から必ず取り込むべき実装要素

先生の報告書は、今回の設計判断の根拠としてそのまま採用してよい。特に以下は mandatory。

* 取得処理と研究本体計算の分離
* JSON-LD ネストのフラット化
* ローカル DB / Parquet の中間保存
* キャッシュ有効期限とバージョン管理
* 参照 ID の事前解決
* 必要なデータだけを SQL / 列指向で読む構成
* 生 JSON の毎回全件ロード禁止 

Tokyu 側 canonical 層は、まさにこの方針を実装する場所として扱うこと。

## 11. 最後の判断基準

この修正が成功したかどうかは、見た目ではなく次で判断する。

* **Tokyu が Toei 並みに “使える feed” として見えるか**
* **営業所・車両・路線の割当が実際にできるか**
* **subset を切って simulation へ流せるか**
* **ODPT 生データの重さを main app に持ち込んでいないか**

ここを満たさない限り、中途半端な UI 修正で終わらせないこと。
