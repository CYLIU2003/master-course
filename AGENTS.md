# AGENTS.md
# Timetable-Driven Dispatch Planning System

## Purpose

This document specifies the architecture and non-negotiable rules for the
EV bus simulation dispatch planning module (`src/dispatch/`).

The central principle is:

> **Timetable first, dispatch second.**

A vehicle dispatch plan must always be *derived from* the timetable, never the
other way around. The timetable is the single source of truth. Dispatch plans
exist to serve the timetable constraints, not to define them.

---

## Core Requirement

Given a set of revenue trips defined in the timetable, produce a set of vehicle
duties such that:

1. Every trip in the timetable is covered by exactly one vehicle duty.
2. Every consecutive trip pair within a duty is physically possible
   (time, location, and vehicle type constraints all satisfied).
3. No trip connection is invented or assumed that violates the feasibility rule.

---

## Feasibility Logic

A vehicle may operate trip **j** immediately after trip **i** if and only if
**all** of the following hold:

### 1. Location Continuity
The vehicle must be able to reach the start of trip j from the end of trip i.

- If `trip_i.destination == trip_j.origin`: no deadhead needed.
- Otherwise: a `DeadheadRule` must exist for the pair `(trip_i.destination, trip_j.origin)`.
  If no rule exists, the connection is **infeasible** — the vehicle cannot teleport.

### 2. Time Continuity (hard constraint)
```
arrival_time(i) + turnaround_time(i.destination) + deadhead_time(i.destination, j.origin)
    <= departure_time(j)
```
If this inequality does not hold, trip j **must never** be connected after trip i.
This is not a soft preference or a penalty — it is a hard infeasibility.

### 3. Vehicle Type Constraint
The vehicle type must be listed in `trip_j.allowed_vehicle_types`.
If not, the connection is infeasible regardless of timing.

---

## Required Processing Pipeline

The pipeline must execute steps in this exact order:

1. **Load timetable** — parse and validate the structured trip data.
2. **Apply feasibility rules** — evaluate every pairwise connection candidate
   against the three constraints above.
3. **Build connection graph** — construct a directed graph where edges represent
   feasible connections only.
4. **Generate dispatch plan** — assign trips to vehicles using the graph
   (greedy or MILP; greedy is the baseline).
5. **Validate output** — confirm every generated duty satisfies all constraints.

---

## Output Requirements

Each `VehicleDuty` in the output must include:

- `duty_id` — unique identifier
- `vehicle_type` — the vehicle type assigned
- `legs` — ordered list of `DutyLeg` objects, each referencing a `Trip` and
  recording the deadhead time from the previous trip

Each `ValidationResult` must include:

- `valid` — boolean
- `errors` — tuple of human-readable error strings (empty if valid)

---

## Validation Standards

All duties generated by any dispatcher (greedy or MILP) must pass
`DutyValidator.validate_vehicle_duty()` before being considered final output.

Validation failures must be surfaced as warnings in `PipelineResult.warnings`
and the affected duty IDs listed in `PipelineResult.invalid_duties`.

---

## Implementation Notes

- `src/dispatch/` must not import from `app/`.
- `src/dispatch/` must not import from `src/constraints/` or `src/pipeline/`.
- `src/dispatch/models.py` dataclasses are intentionally separate from
  `src/schemas/duty_entities.py` to avoid coupling the dispatch layer to the
  MILP solver layer. Import both by full module path if both are needed.
- `hhmm_to_min()` is the canonical time conversion function. All time
  comparisons must use integer minutes from midnight.
- Turnaround rules apply at the *destination* of trip i (the stop where the
  vehicle waits before the next trip).
- Deadhead time is always in the direction (trip_i.destination → trip_j.origin).
  Reverse deadhead times are not assumed to be equal.

---

## Non-Negotiable Rules

1. **No physical impossibilities**: A trip chain that violates the time
   continuity constraint must never appear in any output, at any log level.
2. **Timetable is read-only**: The dispatch layer must never modify or rewrite
   trips, departure times, or arrival times.
3. **`app/` calls `src.dispatch.*` only**: Dispatch logic must not bleed into
   Streamlit pages. Pages call the pipeline; the pipeline owns all logic.
4. **`constant/` is read-only**: Never modify files under `constant/`.
5. **All 138 pre-existing tests must continue to pass** after any change.

## Future Expansion Targets

- Replace `DispatchGenerator` greedy heuristic with a MILP model
  (same interface: `context + vehicle_type → List[VehicleDuty]`).
- Add energy feasibility checks (SoC tracking for BEV duties).
- Add multi-depot support via extended `DeadheadRule` tables.
- Add driver constraint integration (shift limits, break requirements).
- Support multi-vehicle-type duties (mixed-fleet relay).
